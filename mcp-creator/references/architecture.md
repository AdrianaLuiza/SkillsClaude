# MCP Creator — Architecture Reference

## Pipeline Overview

```
API Specs (JSON/OpenAPI/Postman) + enum_types.json
    │ generator.py parses specs + resolves enums
    ▼
Python async tool functions (N auto-generated modules + manual modules)
    │ server.py registers with FastMCP
    ▼
FastMCP server
    │ CodeMode transform (always enabled)
    ▼
3 meta-tools (search, get_schema, execute)
    │ agent uses
    ▼
Claude / any MCP client
```

## Key Design Decisions

### Code Mode — always enabled
Code Mode is always on. It collapses all tools into 3 meta-tools (`search`, `get_schema`, `execute`), providing a consistent interface regardless of tool count. Group tools into logical modules for discoverability via `search`.

### Module grouping
Group specs by domain, not by entity. Example:
- `accounts` (AdAccount, AdAccountCreationRequest)
- `campaigns` (Campaign)
- `instagram` (IGUser, IGMedia, IGComment)

### Spec formats supported
1. **SDK Codegen JSON** (facebook-business-sdk-codegen style): `apis[]` array with endpoint/method/params
2. **OpenAPI 3.x**: paths with operations, parameters, requestBody
3. **Postman Collection v2.1**: items with request objects

### Auth patterns
- Bearer token via env var
- API key via env var
- OAuth2 (token refresh)
- HMAC/appsecret_proof (Meta-specific)

### Manual tools pattern
Some tools need hand-written logic (e.g., targeting search, generic API fallback). Place them in `tools/` alongside generated modules, but add their module names to `skip_modules` in `_register_generated_tools()` so they aren't double-registered. Register them explicitly in `server.py`.

### Enum definitions file
`specs/enum_types.json` contains all enum values referenced by spec parameters. The generator loads this file and injects enum values into function docstrings so agents can discover valid parameter values via `get_schema`. Without this file, enum parameters show as plain `str` with no valid values listed.

### Fields constants
The generator creates a `{ENTITY}_FIELDS` constant per spec listing all readable field names. This powers the `fields` parameter hint in GET endpoint docstrings.

## Dependencies

```toml
[project]
dependencies = [
    "fastmcp[code-mode]>=3.1.0",   # MCP framework + Code Mode (latest patch/minor é puxado em fresh install; rode `uv lock --upgrade-package fastmcp` pra atualizar projetos existentes)
    "pydantic-monty==0.0.7",        # PINNED: 0.0.8 quebra external_functions no CodeMode (fastmcp bug). NÃO atualizar sem rodar smoke test do CodeMode antes — esse pin é o gargalo real do upgrade.
    "httpx>=0.28.0",                # Async HTTP client
    "prefab-ui>=0.10.0",            # Optional: Prefab Apps dashboards
]

[project.scripts]
my-api-mcp = "my_api_mcp.server:main"  # CLI entry point

[tool.hatch.build.targets.wheel]
packages = ["src/my_api_mcp"]
```

### Upgrade workflow do fastmcp

A constraint `>=3.1.0` permite que **fresh installs** peguem a versão mais recente automaticamente. Projetos já scaffoldados ficam congelados pelo lockfile — pra atualizar um existente:

```bash
# uv (recomendado)
uv lock --upgrade-package fastmcp
uv sync

# pip / pip-tools
pip install -U 'fastmcp[code-mode]'
```

**Antes de promover o upgrade**, rode um smoke test do CodeMode (registrar tool + chamar `execute` com `external_functions`). Se aparecer `Monty.__new__() got unexpected keyword argument 'external_functions'`, o pin de `pydantic-monty==0.0.7` virou incompatível com o novo fastmcp — nesse caso, role o fastmcp pra trás (`uv lock --upgrade-package fastmcp --resolution lowest-direct` ou pin manual) até que monty acompanhe.
